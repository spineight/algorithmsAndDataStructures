## Сложность алгоритмов
Логично измерять в количестве операций.
Сколько операций программа совершает зависит от размера входа.

Пример  
Первая программа: 534n^3+12n^2+252 -> n^3  
Вторая программа: 24n^4+10  -> n^4

Проблема оценки времени работы возникает на достаточно больших входах,
на маленьких входах все программы быстро работают.

Оценка:
1) Только худщее слагаемое
2) С точностью до коэффициента


Программы которые работают за n^k - говорят, что они работают за полином (полиномиальное время работы)  
За 2^n - экспоненциальное время работы

log(n) - мы не указываем основание логарифма,   
так как переход от одного основания к другому по свойству 3 - за счет домножения на константу,  
которая не зависит от n


## Логарифмы

### Определение
log_b(a) = x ==> b^x == a
Это такая степень, в которую нужно возвести b, чтобы получить a

### св-во 0
log_a(a^x) = x

### св-во 1
log_a(1) = 0

### сво-во 2
log_a(x*y) = log_a(x)+log_a(y)  
Рассмотрим на примере:  
log_2(32*16)=log_2(2^5 * 2^4) = log_2(2^9) = log_2(2^5) + log_2(2^4)

### св-во 3 (переход на новое основание)
log_2(1024) = 10  
log_10(1024) - ?

Зная логарифм по одному основанию, узнать логарифм по другому  
Давайте изучем как

Воспользуемся log_a(a^x) = x  
Попробуем найти log_b(a^x)  (0),
Давайте a представим в виде b в какой-то степени:
a = b^y, предположим, что нам это удалось, тогда
log_b((b^y)^x) = log_b(b^{xy}) = xy (1)
y = log_b(a)
a^x заменим на z, тогда (1) перепишем как
xy = log_a(z) y
в (0) подставим z=a^x -> log_b(z) = log_a(z) * log_b(a)

Итого: от логарифма по одному основанию можно перейти к логарифму по другому домножив на какое-то число,
Давайте попробуем воспользоваться:
найдем log_8(64) с помощью log_2(64)=6
log_8(64) = log_2(64) * log_8(2) = 6 * 1/3 = 2

log_a(z) * log_b(a) - константа, он от z не зависит.
Логарифмы по разным основаниям друг от друга отличаются в константу

Пример сложение двух чисел в столбик
(кол-во цифер в числе x = ciel(log_10(x)))
x |_|_|_|_|_|
y |_|_|_|_|_|
Поэтому сложность - количество цифер в числе ~ max(log(x), log(y))

#### Правильные скобочные последовательности
Зададим понятие рекурсивно:
1. Пустая скобочная последовательность A - правильная
2. Если A - правильная то, (A) - правильная
3. если А и В - правильные скобочные последовательности, то АВ - правильная

Можно решить используя стэк, если используются скобки одного вида - то можно и без стэка,
для этого храним количество открытых и закрытых скобок.
Если скобки нескольких видов: (), [] , {} - то для решения понадобится стэк
[ ( ] ) - пример когда метод подсчета на работает и нужен стэк

Для скобок нескольких видов, правила такие:
1. Открывающая -> кладем
2. Закрывающая -> извлекаем и проверяем тип
3. В конце стэк должен быть пуст

### Списки
Список - последовательность элементов, которая у нас упорядочена в некотором смысле, а именно:
для каждого элемента известен следующий и предыдущий - двухсвязный (иногда только следующий/предыдущий - односвязный)

Голова  - первый элемент     
Хвост - последний элемент

Поддерживаемые операции:
- добавление в хвост/голову/после какого-то элемента
Отличается от Deque - здесь можно добавлять после заданного элемента за O(1)
Элементы списка будем обозначать номерами (но только эти номера будут идти не по порядку в списке, иначе мы не сможем вставить,
а это будут их вечные номера)

Заведем массивы:
next 
value

Фиктивная голова (первый элемент никаких данных не содержит и является головой и в начале хвостом),
так нам не нужно будет писать проверки на добавление в пустой список,
в таком списке add_first/add_last не нужны, есть только add_after

Списки нужны для того, чтобы по ним ходить

#### Переиспользование освобожденных ID
Если у нас список на массиве длиной k,
то после того как мы его полностью заполним, а потом извлечем все элементы по одному,
то хотя он и будет пустой, мы туда больше не сможем добавить элементов из-за того, что free 
который мы используем для присвоения индетификаторов элементов будет выходить за пределы массива.
Поэтому нужно переиспользовать освобожденные ID
