### Топологическая сортировка за m
В лекции 4 мы изучали способ который работает за n^2 (n - количество вершин)  
(мы брали вершину с inDeg = 0, далее степени inDeg всех вершин куда из нее исходят ребра уменьшали на 1 и опять
брали вершину с inDeg).  
Работает за m - количество ребер  
Количество ребер может быть и n^2 (макс количество),
но бывают ситуациии когда количество ребер как-то искуственно ограничено

Рассмотрим граф, для того чтобы можно было применить топологическую сортировку он должен быть ациклический.  
посмотрим на  вершинки в том порядке, в котором мы из них выходили   
```
               1(1)
            /    \
       2(3)< <----  >3(2)
         |          |
         |          |
         |          |
     4(4)\/ ----->  \/ 6(5)
            \      /
             >   <
               5(6)
```
Занумеруем вершины в порядке как мы из них вышли, только сконца  
в скобочках рядом с каждой вершиной напишем этот номер,  
никакое ребро не ведет из вершины с большим номером в вершину с меньшим.  

```
in 1        Порядок выхода из вершин:
in 2
in 4
in 5
out 5        6
in 6
out 6        5
out 4        4
out 2        3
in 3
out 3        2
out 1        1
```

Оказывается, если мы запустим обход вглубину из какой-то вершины, из которой для всех можно дойти  
и пронумеруем вершины в порядке как мы из них выходили (но правда на оборот), тоесть не  
по возрастанию а по убыванию то получится topSort - топологическая сортировка.  

Давайте разбиремся почему это работает,  
у нас должно какое условие выполняться: из вершины из которой мы вышли раньше,  
нет ребра в вершину из которой мы вышли позже  
Если ориентированный граф ациклический и мы запустили обход в глубину из вершины из,  
которой достижимы все, то не бывает ребра из вершины из которой мы вышли раньше в  
вершину из  которой мы вышли позже, а это то, что как раз нам нужно, что ребер из вершин  
больших в вершины меньшие не бывает.     
Давайте посмотрим, запустим обход вглубину наш, который выводит когда вошли в вершину,  
когда вышли, посмотрим на его аутпут и соединим  моменты входа и выхода вершины (например скобокой) 

```
in 1           --------------------|+1
in 2           ---------------|+2   |
in 4           ---------| +4  |     |
in 5           -|+5  |  |     |     |
out 5        6 -|-5  |  |     |     |
in 6           -|+6  |  |     |     |
out 6        5 -|-6  |  |     |     |
out 4        4 ---------| -4  |     |
out 2        3 ---------------|-2   |
in 3           -|+3                 |
out 3        2 -|-3                 |
out 1        1 ---------------------|-1
```
Мы уже вчера обсуждали с вами, что всегда будет типа правильной скобочной последовательности  
Система вложенных отрезков, отрезки либо не пересекаются, либо один внутри другого.  
Это логично потому что это моменты входа/выхода в рекурсию, как они могут пересечься.  

Вот предположим, от противного, что у нас найдется ребро в вершину из которой мы вышли раньше,  
из вершины из которой мы вышли позже  
Тогда сейчас мы получим противоречие, к сожалению тут доказательство довольно сложное,  
два случая, поэтому приготовьтесь.  
Итак:  
Вот мы вышли из вершины a, позже чем из вершины b:

Рассмотрим сначала вложенный случай

Мы дошли в вершину b, когда еще не вышли из вершины a, значит у нас есть путь из  
вершины a~b в вершину b  
У нас вершинка a была серой (когда мы вошли в вершину, она стала серой, когда выходим из нее становится черной)  
А серые вершинки образуют путь от начала, тоесть у нас от 1 есть путь, где-то на нем лежит вершинка a и дальше  
он ведет в вершинку b
```c++
      1
      |
      |
      |
      V
      a <--/
      |   /
      |  /
      V/
      b
```  
И у нас есть, мы утверждаем ребро из b->a, это противоречит тому, что граф ациклический  
Рассмотрим случай, когда отрезки невложены:
```c++
---| +b
   |
   |
---| -b

---| +a
   |
   |
---| -a
```
И предположим есть ребро из b в a  
В момент, когда мы вышли из b - в a еще не заходили, а из b есть ребро в a  
Значит кто-то не умеет писать обход вглубину  
Получается, такой ситуации быть не может  
Значит наше утверждение верно  


 

```c++
dfs(u) {
    vis[u] = 1;
    std::cout << "in" << u << '\n';
    for(v сосед u) {
        if(!vis[v]) {
            dfs(v)
        }
    }
    std::cout << "out" << u << '\n';
}
```

Если мы запишем вершины как мы из них вышли, только в обратном порядке -
у нас получится топсорт.
Разберемся почему это работает:
У нас должно выполняться условие - из вершины из которой мы вышли раньше,
нет ребра в вершину из которой мы вышли позже.


"запустили обход вглубину из вершины из которой допустимы все"  
Проблема, как найти вершину из которой достижимы все  
Можно вылечить следующим образом:  
 допустим есть какой-то граф и внем может быть и есть вершина из которой все достижимы,
 а может быть и нету, даже если есть - непонятно как ее найти.  
 Давайте добавим в этот граф еще одну вершину, назовем ее 0 и из нее проведем ребра во все  
 (это не может добавить цикл, так как цикл не может прийти через вершину 0 ему не через что туда зайти, а внутри графа
 циклов и не было)  
 У этой вершины inDeg+ = 0 и из нее достижимы все вершины графа  
 Давайте посмотрим как будет выглядеть DFS запушенный из вершинки 0  
    он пробегает всех соседей из вершины 0
```
for(int i = 1; i <= n; ++i) {
    if(!vis[i])
        then DFS(i)
}
```
Ну так это тоже самое, что вместо того, чтобы добавлять эту вершину 0
по очереди запуститься из всех вершин, которые белые после предыдущих  
Никуда не надо записывать, эту искуственную вершину, просто в нашей программе,   
когда мы пишем топологическую сортировку мы пишем на DFS(0) - а код:  
```
for(int i = 1; i <= n; ++i) {
    if(!vis[i])
        then DFS(i)
}
```  
Что просто мы как бы виртуально добавили этот нолик и по всем его соседям пробежались  
Тоесть внешний самый запуск процедурки DFS мы расписали  


Этот алгоритм работает за один запуск DFS  
(Можно это рассмотреть как один большой DFS из нуля(искуственной вершины))  
За сколько работает DFS, нужно рассмотреть два случая, как мы храним граф:  
   * для матрицы смежности - `n^2` (n раз мы заходим в процедуру DFS, она выполняется за n - время выяснения соседей)
   * для списков смежности - `m` (количество ребер) (`n` раз мы заходим в процедуру DFS, она выполняется за DegU-, вызывается она для всех вершин, поэтому получается: `Deg1-+Deg2-+...Degn- - E`)

Если мы топологически попытаемся отсортировать граф в котором есть цикл,  
рассмотрим этот цикл:

```c++
    x0
    <----- ^ x3
    |      |
    |      |
    V------> x2
    x1
```
Вершины с меньшими номерами, должны вести в вершины с большими:  
`x3> x2 > x1 > x0`
Но `x3` ведет в x0, и номер у него должен быть меньше, но он получается больше - это неправильно    


#### Выявить что есть цикл в графе.
Два способа
1. На базе топологической сортировки   
время работы m (позволяет определить факт цикла, но сам цикл найти не позволяет)
Проведем топологическую сортировку нашим алгоритмом  
Вершины занумеруются в каком-то порядке,  

Посмотрим на все ребра снова, мы знаем, что  
- Если граф был ациклический, то построена топологическая сортировка
- Если граф не ациклический, то у него не существует топологической сортировки

Как с помощью этих утверждений проверить граф на ацикличность:  
Смотрим на соседей вершины u - если найдется сосед v для вершины u c номером  
так что номер v < номера u - значит есть цикл

```c++
     / 
  u  --
    \
```

2.
Модифицируем алоритм обхода в глубину, чтобы он не просто искал есть ли цикл,
но еще и находил этот цикл.  
Введем массив color:  
`0` - белые  
`1` - серые  
`2` - черные  

```c++
  dfs(u) {
    color[u] = 1
    push(u)
    for(v сосед u) {
      if(found) return;
      if(color[v] == 0)
        dfs(v)
      if(color[v] == 1) {
        цикл - вывести цикл (нужно стэк вывести в обратном порядке)
        found = true;
        return;
      }
    }
    pop(u)
    color[u] = 2
  }
```
В графе есть цикл, тогда и только тогда когда мы пытаемся пойти из текущей серой вершины в серую  
Мы нашли факт существования цикла, как же найти сам цикл  
На входе в dfs мы добавляем u в стэк, когда делаем ее черной - вынимаем из стэка  
если есть цикл - мы знаем что есть обратное ребро u-->v, в стэке будут серые вершины (серый путь)  

```
| |
|u| - верхушка стэка
|#| \
|#|  \
|#|   |    --- цикл
|#|  /
|v|<--
| |
| |
```
Начинаем вынимать из стека вершины, пока не найдем v  


Последний аспект поиска цикла: что делать если граф не ориентированный?  
К сожалению вот это так напрямую совсем сделать неправильно, потому, что  
у любой вершины, кроме одной есть всегда серый сосед - предыдущая (тот из которой мы в нее пришли)  
Для этого добавляем условие - назад не смотрим, можно передавать в качестве параметра parent

```c++
  dfs(u, parent) {
    color[u] = 1
    push(u)
    for(v сосед u) {
      if(found) return;
      if(color[v] == 0)
        dfs(v,u)
      if(color[v] == 1 && parent[v] != u) {
        цикл - вывести цикл (можно стек не переворачивать, у нас граф не направленный)
        found = true;
        return;
      }
    }
    pop(u)
    color[u] = 2
  }
```

### Кратчайшие пути в графах
Обход в глубину позволял нам добраться из одной вершины в другую, найти путь есть он или нет,
ну и сам путь можно найти.  
Давайте попробуем на этом графе найти путь из 1->4  
```
        1
       / \
      2   4
       \ /
        3
```
Обход в глубину пойдет(может пойти):  
`1->2->3->4`  
Путь нашли, но это не самый лучший путь.  
Поиск кратчайших путей - достаточно важная задача, во многих других задачах используется в качестве подзадачи.
И сама по себе достаточно ценна.  

Мы рассмотрим целое семейство таких алгоритмов, которые применимы в разных ситуациях.

Начнем с самой простой задачи, задача такая:  
дан граф, ориентированный или нет - это не важно,  
длина пути, это как мы договаривались, количество ребер в нем,    
есть вершинки две `s,t` : `s` - откуда мы хотим идти, `t` - куда (начало и конец пути)  
Найти путь из s~~~~>t, который содержит минимальное количество ребер,  
тоесть мы считаем, что проход по ребру занимает единицу времени/стоимости и хотим минимизировать стоимость
прохода по пути.  

Первый метод - волновой алгоритм  
Максимальное расстояние в графе с n вершинами = n-1 (n-1 ребро)  
`-->-->-->-->`
```c++
for (int i = 1; i <= n; ++i)
    d[i] = INF // n+1 или n - точно достаточно, ясно что путь будет короче
d[s] = 0
for(int i = 0; i <= n-1; ++i)
    for(int j = 1; j <= n; ++j) { // перебераем все вершины
        if(d[j] == i) // если до j-ой вершины расстояние == i
            for(k сосед j) {
                if(d[k] > d[j]+1) {
                    d[k] = d[j] + 1
                }
            }
    }
```
Время работы алгоритма:
 * `n^3` матрица смежности
 * `nm` (для списков смежностей)  
 
Он делает кучу безсмысленных действий:  
Когда мы просматриваем вершину и if не выполняется:  
( ```if(d[j] == i) // если до j-ой вершины расстояние == i```)  

Давайте подумаем, нельзя ли как-то этот алгоритм улучшить, чтобы он просматривал только те вершины, которые надо.
```
|s|        |         |         |
 0    1         2         3             - расстояние
```
Заметим, что вершины на расстояние 2 берутся во время рассмотрения вершин на расстояние 1  
Вершины на расстояние 3 берутся из вершин на расстояние 2 и т д.  
вершины на расстояние к+1 берутся из вершин на расстояние к  

В начале для s расстояние = 0, для остальных вершин - неизвестно  
Заведем очередь и положим в нее сначала вершины на расстояние 0  
Извлекли вершинку на расстоянии 0 далее посмотрим на ее соседей - они все на расстояние 1, добавим их в очередь  

Таким образом берем вершину из очереди - она на расстояние к, ее соседи которых мы не видили раньше будут на к+1  
(мы кладем в очередь вершину только если расстояние до нее пока неизвестно)  
В очереди все время будет поддерживаться: 
```
|       | k   | k + 1 |`
```   

Этот алгоритм называется обход в ширину BFS (Breadth first search - сначала идем в ширь)  
Наша программа постепенно узнает расстояние до вершин, сначала у нас известно расстояние только для стартовой вершины      
``` c++
for(int i = 1; i <= n; ++i)
    d[i] = -1; // или INF
d[s] = 0
queue q;
q.push(s);
while(!q.empty()) {
    int u = q.top(); q.pop();
    for(v сосед u) {
        if(d[v]==-1) {
            d[v] = d[u] + 1
            q.push(v);
        }
    }
}
```
Доказательство что так мы можем найти минимальное расстояние до вершины t:  
Предположим что для какой-то вершины нашлось неправильно  
Выбирем среди этих вершин, вершину u с минимальным истинным расстоянием `x[u]`  
Два случая:  
`1). x[u] < d[u]   // d[u] - это то расстояние которое мы нашли`  
Рассмотрим самый короткий путь от s до u :  

s ~~~~~~~~~~~v-->u  
на нем есть предыдущая вершина v(поскольку она ближе чем u, и u у нас вершина с минимальным истинным расстоянием, для которой мы неправильно нашли, то для v мы правильно нашли)  
для нее `d[v] = x[v]` // найденное расстояние == истинному  
тогда получается противоречие: когда v извлеклось из очереди мы должны были пойти по ребру  
`v->u и d[u] = d[v]+1` - но мы почему-то этого не сделали  

2). `x[u] > d[u]`
Рассмотрим тогда вершину из которой мы пришли, когда сделали:    
`d[u] = d[v]+1`, это вершинка v, рассмотрим вершину из которой мы пришли когда для v это сделали  

и так далее, так мы дойдем до s
Каждый раз d будет уменьшаться на 1, мы получим путь `d[u]`, он получился короче чем `x[u]`  
Очень подозрительно, такого быть не может        

Мы пока научились находить только длину, давайте находить сам путь:  
давайте для каждой вершинки запомним а откуда в нее лучше всего прийти (`from`)

``` c++
for(int i = 1; i <= n; ++i)
    d[i] = -1; // или INF
d[s] = 0
queue q;
q.push(s);
while(!q.empty()) {
    int u = q.top(); q.pop();
    for(v сосед u) {
        if(d[v]==-1) {
            d[v] = d[u] + 1
            q.push(v);
            from[v] = u;
        }
    }
}
```

``` c++
print(u)  {// выводит путь от s до u
    if(u != s) print(from[u]);
    std::cout << "u";
}
```
Время работы поиска в ширину - m (списки смежности)  
Паталогический случай, если в графе меньше ребер, чем вершин существенно: `O(m+n)`    

Предположим что у нас в графе ребра не все одинаковые а есть длинные и короткие
(по длинным идти 2, а по коротким 1)  
Как решить эту задачу используя обход в ширину:  
В ребра длины 2 добавим вершину в серидину - получим все ребра длинной 1  


### Пути во взвешенных графах
Граф: множество вершин и множество ребер которые как-то соединяют эти вершины,  
взвешенный граф - граф где каждому ребру сопоставлен какой-то вес (ну или длина/стоимость)

Вопрос как хранить вес для ребер:  
Для матрицы смежности можно хранить вес, а 0- будет значить нет ребра,  
либо две матрицы - смежности + весов  
Для списков смежности:  
```
dest |                              |
w - заведем доп массив, вес нашего ребра
next |                              |
      1 2 3 ....                   m
head |              |
      1 2 3 ...    n
``` 

В задачах на поиск кратчайшего пути, довольно типичная ошибка - сделать либо слишком большую либо слишком маленькую бесконечность  


### Алгоритм Дейкстры (Dijkstra)
Работает за `n^2`  
И решает задачу о кратчайших путях из одной вершины для взвешенного неотрицательного графа (веса не отрицательные)

У нас корректность обхода в ширину для невзвешеннго графа основывалась на том, что вершинки из очереди достаются
в порядке возростания растояния до них и кладутся в этом порядке.  
Обычная очередь нам теперь не подходит, нужно, на самом деле, использовать приоритетную очередь,
не будем использовать сложную Heap реализацию, мы будем использовать простую(наивную) реализацию приоритетной очереди -
приоритетная очередь на массиве.    

Алгоритм Дейкстры действует:  
Все вершины делятся на два множества: помеченные и не помеченные,  
помеченные U - до них нам уже известно правильное расстояние.  
Шаг алгоритма Дейкстры - берется непомеченная вершина у которой минимальная текущая гипотеза расстояния  
у каждой вершины хранится в массиве `d[u]` - за сколько мы умеем до нее доходить (такая гипотеза)  
она помечается и по всем ребрам которые ведут из этой вершины проводится операция которая называется - релаксация,  
мы смотрим, а что лучше то как мы раньше умели ходить до этой вершины или через вершину u и минимум из этих двух
запоминаем.  
 
```c++
В начале: множество U - пусто, расстояние до всех вершин = бесконечность    
d[s] = 0  
while(пока есть непомеченные вершины достижимые из s)
  u<- вершина такая что marked[u] = false && d[u] - min  
  marked[u] = true
  for(v сосед u ) {
    релаксация d[v] значениями d[u] + weight(uv)  
}
```
